# Paipu Player Future Roadmap

## Summary

这份文档记录牌谱播放器后续 backlog，按当前优先级从高到低排列。目标是把已经能读取和播放真实雀魂牌谱的原型，逐步打磨成更接近游戏内观感、适合复盘和提问分析的工具。

本文件只记录方向、依赖关系和初步拆分，不要求一次实现完。后续每一项都可以再拆成独立 agent 文档或实现计划。

## Current Facts

- 当前 normalized `Round` 已包含 `doraIndicators`，但 UI 还没有把宝牌指示牌作为稳定区域展示出来。
- 当前回放状态能读取 `RecordDealTile`，也就是每一次实际发生的摸牌事件，并带有 `leftTileCount`。
- 当前还没有暴露“完整牌山/未摸牌山”。因此后续分支推演只能先基于已知摸牌序列做实验，不能假装已经拥有完整未揭示牌山。
- 当前牌张主要以文本/样式块显示，后续可以替换为图片资源，并保留文本 fallback。
- v2 当前核心范围仍是四麻普通规则牌谱展示；AI 分析和分支推演属于后续能力扩展。

## Priority Backlog

### P1. 手牌排序

目标：每家的手牌按稳定麻将顺序展示：万子、筒子、索子、字牌；同种类从小到大排列。字牌顺序为东、南、西、北、白、发、中。

初步拆分：

- 在数据层或播放状态层提供统一 `sortTilesForHand` 工具，避免每个组件各自排序。
- 目标玩家手牌和后续可能展示的其他玩家手牌都使用同一排序规则。
- 摸入牌是否单独放在最右侧需要在实现时明确：默认推荐“手牌主体排序，当前摸牌仍可单独高亮在右侧”，更接近真实出牌决策体验。

验收要点：

- 初始手牌、摸牌后手牌、吃碰杠后手牌都保持稳定排序。
- 红五或特殊编码不会破坏排序。
- 测试覆盖万/筒/索/字混合、重复牌、非法或未知牌值 fallback。

### P2. 显示宝牌指示牌

目标：在牌桌中央或局信息区域明确显示当前局宝牌指示牌。

初步拆分：

- 使用现有 `Round.doraIndicators`，先展示开局宝牌指示牌。
- 如果后续 record 中能稳定识别杠后新宝牌，再扩展为随事件进度显示当前已翻开的宝牌指示牌。
- UI 文案使用“宝牌指示牌”，避免和实际宝牌混淆。

验收要点：

- 每局切换后宝牌指示牌正确更新。
- 没有数据时显示空状态，不挤占或破坏牌桌布局。
- debug 模式和普通模式都不重复显示同一块信息。

### P3. AI 分析聊天

目标：在任意暂停的事件点，用户可以在聊天框提问，例如“这一步怎么打”“怎么分析对手情况”“现在危险牌有哪些”。

初步拆分：

- 先定义发送给分析服务的局面快照：牌谱来源、局号、事件 cursor、玩家信息、分数、宝牌指示牌、目标玩家手牌、各家牌河、副露、立直状态、当前事件。
- UI 增加复盘聊天面板，问题和回答绑定当前 round/cursor，避免切换事件后上下文混乱。
- 第一版可以使用结构化摘要或 LLM，不接 Mortal/akochan；后续再接专业麻将引擎。

验收要点：

- 在不同事件点提问时，服务端收到的局面快照不同。
- 回答中明确基于“当前可见信息”，不要泄露未来事件，除非用户开启全牌谱复盘模式。
- API 不返回账号凭据、account_id 或完整 raw records。

### P4. 使用牌图替代文本牌

目标：引入麻将牌图片资源，替代当前文本牌块，让牌桌更直观。

初步拆分：

- 选定一套可合法使用的牌图资源，放入 `public/` 下的稳定路径。
- `Tile` 组件新增图片渲染模式，保留文本 fallback。
- 继续支持红五、当前牌、摸切、立直宣言牌、副露来源等状态样式。

验收要点：

- 图片加载失败时仍能用文本显示牌值。
- 桌面和移动端牌图尺寸稳定，不造成横向溢出。
- 构建产物不引入过大的无关资源。

### P5. 同牌高亮

目标：鼠标或触控指向任意一张牌时，牌桌上所有相同牌同步高亮，帮助用户判断这张牌在哪里出现过。

初步拆分：

- 在牌桌上层维护 `hoveredTile` 状态，由所有 `Tile` 组件共享。
- 手牌、牌河、副露、宝牌指示牌都参与高亮。
- 移动端用点击切换高亮，再点空白处取消；桌面端用 hover。

验收要点：

- 指向 `2z` 时，所有 `2z` 同步高亮，其他牌不高亮。
- 高亮不会影响当前事件、立直牌、摸切牌等原有状态的可辨识度。
- 键盘或无鼠标场景不影响正常播放。

### P6. 游戏式牌桌视觉

目标：逐步减少“每个框框里面放着牌”的面板感，让牌桌尽可能接近真实游戏牌桌的空间体验。

初步拆分：

- 桌面端强化四方位牌桌：玩家名、点数、牌河、副露围绕中心局信息摆放。
- 中央区域保留场风、本场、供托、宝牌指示牌、当前事件等关键复盘信息。
- 移动端不强行复刻完整四方桌，仍以目标玩家和当前事件可读性为第一优先。
- 这一项建议在 P1-P5 稳定后进行，避免视觉重构和基础交互同时变化。

验收要点：

- 桌面端第一眼能看出四家座位关系。
- 牌河、手牌、副露不被卡片边框割裂。
- 移动端仍不出现牌张、按钮、玩家名互相覆盖。

### P7. 分支推演和 AI 续打

目标：允许用户在任意一个回合选择与牌谱不同的行动，然后由 AI 继续打完剩下的牌局，用来探索这一局麻将的其他可能性。

初步拆分：

- 先定义实验范围：第一版只支持从目标玩家某次摸牌后选择不同弃牌。
- 需要规则引擎校验合法动作：切牌、吃碰杠、立直、和牌、流局等。
- 需要 AI 出牌策略或外部引擎，至少能为其他三家和目标玩家后续动作给出合理选择。
- 需要明确牌山来源：当前只能稳定获得已发生摸牌序列，完整未揭示牌山尚未暴露。实验版可以使用原牌谱后续摸牌序列作为“固定未来牌序”，但 UI 必须标注这是模拟假设。

验收要点：

- 用户做出不同弃牌后，原牌谱回放和模拟分支清楚分离。
- 模拟结果明确标注为“基于假设的分支”，不和真实牌谱结果混淆。
- 规则非法动作有明确提示。
- 不在这一阶段承诺完整线上对局级 AI 强度。

## Recommended Phases

### Phase A: 基础可读性

- 完成 P1 手牌排序。
- 完成 P2 宝牌指示牌展示。
- 推荐优先级最高，改动范围小，能快速提高复盘可读性。

### Phase B: 牌张表现

- 完成 P4 牌图替代文本牌。
- 完成 P5 同牌高亮。
- 这两项都集中在 `Tile`、牌河、副露和牌桌交互层，适合一起设计视觉状态。

### Phase C: 牌桌视觉重做

- 完成 P6 游戏式牌桌视觉。
- 依赖 Phase A/B 的牌张组件稳定后再做，减少返工。

### Phase D: AI 分析聊天

- 完成 P3 AI 分析聊天第一版。
- 先接“当前局面结构化快照 + 聊天 UI + 服务端分析接口”，专业麻将引擎可后置。

### Phase E: 分支推演实验版

- 开始 P7 分支推演。
- 先做受限实验版，不承诺完整牌山、不承诺完备 AI 牌力。

## Agent Split Suggestions

- Tile Data Agent：负责排序工具、宝牌指示牌数据接入、相关测试。
- Tile Visual Agent：负责牌图资源、`Tile` 组件图片模式、同牌高亮状态。
- Table Experience Agent：负责游戏式牌桌布局和移动端可读性。
- AI Analysis Agent：负责局面快照、聊天 API、聊天面板和隐私边界。
- Simulation Research Agent：负责分支推演可行性、牌山/摸牌序列边界、规则引擎选型。

## Shared Test Plan

- 每个实现阶段至少运行 `npm.cmd run test` 和 `npm.cmd run lint`。
- 影响 UI 的阶段运行 `npm.cmd run build`。
- 使用 `http://localhost:3000/?debug=1` 和示例牌谱手动验收：

```text
https://game.maj-soul.com/1/?paipu=260429-1ccbed45-15bd-4708-85d3-fabeac0241f0
```

- 移动端验收需要确认不出现横向溢出、控件重复、牌张和玩家名重叠。
- AI 和分支推演阶段需要额外检查不会泄露账号凭据、account_id 或完整 raw records。

## Assumptions

- 优先级按用户给出的编号执行：P1 最高，P7 最低。
- 当前阶段只记录路线，不直接实现功能。
- 后续继续使用 React、Tailwind、lucide-react 和现有 Next 16 结构，不为这些功能提前引入新的 UI 组件库。
- 分支推演在没有完整牌山能力前，只能作为“基于假设的模拟”，不能视为真实牌谱的唯一替代发展。
